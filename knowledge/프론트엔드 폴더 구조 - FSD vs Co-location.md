프론트엔드 폴더 구조는 **규모와 의존성 복잡도**로 가른다. 핵심: 작게 시작하면 **Co-location**, 도메인 의존성이 꼬이면 **FSD** 개념을 점진 도입. 둘은 _물리적 응집도_ ↔ _의존성 격리_ 의 **==트레이드오프==** 관계다.

## Feature-Sliced Design (FSD)
코드를 `레이어(Layers) → 슬라이스(Slices) → 세그먼트(Segments)`의 3단계로 분할하는 규격화된 방법론. 핵심은 **==단방향 의존성 규칙==** — 상위 레이어는 하위를 `import`할 수 있지만 역방향·동일 레이어 간 참조는 금지.

### 7대 레이어 (상위 → 하위)
| 레이어 | 역할 | 예시 |
| --- | --- | --- |
| `app` | 전역 설정·프로바이더·라우터 (진입점) | — |
| `pages` | 화면(Route) 단위 구성 | — |
| `widgets` | `features`+`entities` 조합한 거대 독립 UI 블록 | Header, Sidebar |
| `features` | 유저 행동·상호작용 (**동사 중심**) | `add-to-cart`, `search` |
| `entities` | 비즈니스 도메인 데이터 모델 (**명사 중심**) | `user`, `product` |
| `shared` | 비즈니스 로직 없는 순수 공통 모듈 | UI 원자 컴포넌트, 유틸 |

> [!info] `processes` 레이어는 deprecated — 여러 페이지에 걸친 복잡한 프로세스용이었으나 현재는 거의 안 씀.

**장점** — 단방향 규칙 덕에 하위 계층 수정이 상위로 역전파되지 않아 사이드 이펙트를 **구조적으로 격리**한다.
**한계** — 기능 하나(예: 회원가입)가 여러 레이어로 흩어져 **물리적 응집도가 낮고**, "어느 계층에 둘지" 고뇌로 **러닝 커브·설계 병목**이 크다. → 실무 도입의 가장 큰 걸림돌.

> [!warning] 격리 효과는 **규칙이 지켜질 때만** 유효하다. 휴먼 에러로 단방향 규칙이 깨지면 오히려 예측 못 한 문제가 생긴다.

## Co-location (도메인 중심 / 기능별)
_"함께 변경되는 코드는 같은 위치에 둔다"_ 는 철학으로, 도메인 중심으로 컴포넌트·훅·스타일·API를 **한 폴더에 응집**시키는 대중적 패턴.

```text
src/
└── features/ (or domains/)
    └── product/                    # Product 도메인 폴더
        ├── _components/            # 도메인 전용 UI 컴포넌트
        ├── _hooks/                 # 비즈니스 로직 커스텀 훅
        ├── _styles/                # 전역 오염 없는 전용 스타일
        ├── _api/                   # 서버 요청 API 함수
        └── index.ts                # Public API Entry (외부 노출 제어)
```

**장점** — 한 도메인의 모든 것이 한 폴더에 모여 직관적, 초기 개발 속도 빠름, 러닝 커브 낮음.
**한계** — 도메인 간 참조 규제가 약해, 규칙을 강제하지 않으면 의존성이 꼬이기 쉽다.

> [!tip] `index.ts`를 ==Public API barrel export==로 강제하면 내부 구현(`_` 프리픽스)을 감추고 노출 경로를 한곳으로 모아 **최소한의 캡슐화**를 유지한다.

## 비교 & 선택 가이드
| 비교 항목 | Co-location | FSD |
| --- | --- | --- |
| 적합한 규모 | 소형 ~ 중형 | 대형 ~ 초대형 |
| 물리적 응집도 | **높음** | **낮음** |
| 의존성 격리 | 낮음 | **높음** (단방향 강제) |
| 초기 개발 속도 | **빠름** | 느림 |
| 러닝 커브 | 낮음 | 높음 |

- [!] **속도·생산성이 우선이면**: FSD를 무리하게 적용하지 말고 **Co-location으로 시작**하되 `index.ts` barrel export로 최소 캡슐화 유지.
- [!] **도메인 의존성이 꼬였다면**: 전체 FSD 대신 `shared`·`entities` 개념만 **부분 차용**해 점진적으로 결합도를 낮춘다.

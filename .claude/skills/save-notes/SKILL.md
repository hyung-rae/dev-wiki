# save-notes — 대화를 atomic 노트망으로 저장

사용자가 대화 내용을 보관함에 저장해달라고 하면, **한 덩어리 파일이 아니라 개념 단위로 잘게 쪼갠 여러 노트**를 만들고 태그와 `[[링크]]`로 서로 연결한다.

## 절차

### 1. 무엇을 저장할지 범위 정하기
- 직전 답변만인지, 최근 주제 전체인지 모호하면 **한 줄로 확인**한다.
- 저장 가치가 있는 **독립적 개념**들을 추려낸다. 잡담·진행성 대화는 제외.

### 2. 개념 단위로 분해 (원자성)
- **노트 1개 = 개념 1개.** 한 노트에 여러 주제를 섞지 않는다.
- "이건 나중에 이 개념만 따로 찾고 싶을까?"가 기준. 그렇다면 별도 노트로 분리.
- 한 번의 저장에서 보통 1~5개의 노트가 나온다.

### 3. 분류 — 어느 폴더로?
- **버그/장애 해결**성 내용 → `트러블슈팅/`, `템플릿/트러블슈팅.md` 구조 사용.
- 그 외 **일반 지식/학습/개념** → `knowledge/`, `템플릿/knowledge.md` 구조 사용.

### 4. 기존 노트와 중복·연결 점검 (중요)
- 새 노트를 쓰기 전에 `knowledge/`와 `트러블슈팅/`를 훑어 **이미 있는 개념인지** 확인한다.
  - 이미 있으면: 새로 만들지 말고 기존 노트를 **보강**하거나, 관련 있으면 `[[링크]]`만 추가.
- 새로 만든 노트들끼리, 그리고 관련된 기존 노트와 **양방향으로 `[[링크]]`** 를 건다.

### 5. 각 노트 작성 규칙
- **파일명**: 개념을 드러내는 간결한 제목 (예: `knowledge/React useEffect 의존성 배열.md`). 개념 언어를 그대로 따른다.
- **frontmatter**:
  - `tags`: `knowledge`(또는 `troubleshooting`) + **도메인 태그**(예: `react`, `postgres`, `network`).
  - `created`: 오늘 날짜 `YYYY-MM-DD`.
  - `source`: 어디서 나온 내용인지(대화/웹검색 등).
- **본문**: 핵심 결론을 먼저, 그다음 부연. 개념 하나에 집중.
- **`## 관련 노트`**: `[[다른 노트]]` 링크 모음.

### 6. 마무리 보고
- 만든/수정한 파일 목록과 각 노트가 어떤 개념을 담았는지, 어떻게 링크했는지 **간단히 요약**한다.

## 원칙
- 잘게, 그러나 의미 있게 — 너무 잘라 파편이 되거나, 너무 뭉쳐 검색이 안 되는 것 둘 다 피한다.
- 링크 없는 고립 노트를 만들지 않는다. 최소 1개 이상 관련 노트/태그로 연결되게 한다.
- 커밋은 사용자가 요청할 때만. 요청 시 `CLAUDE.md`의 gitmoji 컨벤션(`📝 Docs` / `➕ Add`)을 따른다.
